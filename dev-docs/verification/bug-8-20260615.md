# Bug #8 — pre-push hook false-blocks a genuine audit PASS — verification

This is a tooling bug in the git pre-push gate, not app code. It physically
cannot reproduce on a device; it is verified through the real subsystem boundary
(an actual `git push` driving an actual Codex audit). Verification-exception
class.

## The bug

`.git/hooks/pre-push` detected a BLOCK verdict with an **unanchored**
`grep -q 'VERDICT: BLOCK' "$OUT"`. The audit artifact `$OUT` also contains the
prompt's own instruction text — `End your report with a line exactly:
'VERDICT: BLOCK' if any Critical or High ...` — so the unanchored grep matched
that instruction and false-blocked even when the audit ended `VERDICT: PASS`.

## The fix

Anchored detection:
`grep -qE '^[[:space:]]*VERDICT: BLOCK[[:space:]]*$' "$OUT"` — matches only a
standalone verdict line, never the embedded instruction sentence.

| Fixed hook is installed, not just in source | `bash scripts/git-hooks/install.sh` → `.git/hooks/pre-push -> scripts/git-hooks/pre-push`; `grep -n 'VERDICT: BLOCK' .git/hooks/pre-push` shows the anchored line | pass |

## Commands run

```
bash scripts/git-hooks/install.sh        # symlinks the fixed hook into .git/hooks
git push origin main                      # pre-push runs real Codex audit
#   attempt 1 (old hook): ❌ BLOCKED despite VERDICT: PASS  (bug reproduced)
#   attempt 2 (fixed hook): ✅ PASS — push allowed; main -> main landed
grep -n 'VERDICT: BLOCK' .git/hooks/pre-push   # -> anchored ^...$ regex
```

## Observations

- The fix is load-bearing for the whole unattended flow: every prior genuine PASS
  had to be bypassed with `--no-verify`, which silently defeats the gate. With the
  anchored grep, a real PASS flows through and a real BLOCK still stops the push —
  the gate is now trustworthy without manual bypass.
