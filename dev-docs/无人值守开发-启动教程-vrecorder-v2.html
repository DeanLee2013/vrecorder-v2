vrecorder-v2 无人值守开发 · 启动教程

这是 vrecorder 的干净重启版。治理骨架已全部就位，且多了一台 vrecorder 当初没有的引擎——提交即审计闸门（pre-push Codex 审计，从第 1 个 commit 起生效）。本教程给你正确的提示词与启动顺序：冷启动 → 设计 → 计划 → 六门开发 → 提交即审计 → 无人值守闭环。

01你现在站在哪里

vrecorder-v2 不是从白纸开始的——它带着 vrecorder/vmark 已验证的整套纪律落地，外加一台新引擎。

✅ 一句话定位：地基已浇好，且第 1 个 commit 就装了"出门安检"

vrecorder 当初的痛点是"门禁很多、引擎从未点火"。vrecorder-v2 直接把那台当初最该有却没有的引擎装上了：提交即审计——每次 git push 前自动请独立的 Codex 审一遍改动的 Swift，严重问题挡在本机门口，根本推不出去。

· 骨架实测在位：.claude/rules/ 13 条、hooks/ 6 个治理钩子、cron-prompts/ 4 份、scripts/{run-codex,run-tests}.sh、.claude/agents/ 9 个子代理（嫁接自 vmark，待 iOS 改造）。

· 审计引擎可用：codex CLI 走 ChatGPT 订阅登录（无 OpenAI key）；scripts/git-hooks/pre-push 已 install.sh 装入 .git/hooks/。

· 记忆框架在位：dev-docs/{audit/DIMENSIONS-ios.md, verification/SCHEMA.md, decisions/, plans/, designs/}。

📌 与 vrecorder 的关键差别：vrecorder 的审计靠 cron/Codex 在工作流中段跑（且当初从未点火）；vrecorder-v2 把审计前移到每次推送的物理闸门——这是"重启"的全部意义。先把这台引擎吃透（§03），再谈五阶段。

02一次性点火序列

开工前做这 5 件事，之后不再重复。

1确认审计引擎登录态（订阅，非 key）
codex login status        # 期望: Logged in using ChatGPT
codex --version           # 任何 0.13x+ 均可
2装好提交即审计钩子（若 clone 来的新机）
bash scripts/git-hooks/install.sh   # 装 .git/hooks/pre-push
3在仓库根启动 Claude Code，让它读懂项目
提示词 · 让助理认识 vrecorder-v2读 AGENTS.md、docs/architecture.md、dev-docs/audit/DIMENSIONS-ios.md。用要点回答：
(1) 这个项目的六门工作流是什么、每门的验收 bar；
(2) "提交即审计"闸门的触发点、阻断条件、绕过方式；
(3) 现在处于哪个里程碑、下一步该做什么。不要复述全文，给我可执行结论。
4确认免确认授权（cron 会话不能停下来等你点同意）

把常规工具调用预先放行，否则 cron 唤起的会话会卡在权限弹窗上。治理钩子是exit 2硬挡的只在该挡时挡；提交即审计闸门用 --no-verify 可显式绕过。

5准备远程仓库（已就绪）

已建 DeanLee2013/vrecorder-v2（private），main 已有首个 commit。后续每次 git push 自动过闸门。

03核心差异引擎：提交即审计

这是 vrecorder-v2 的灵魂，先单独讲透，再嵌进开发流。

git push改动的 Swift 即将离开本机
→
pre-push 钩子取本次推送 diff 里的 *.swift
→
run-codex.sh订阅版 Codex 只读审计
对照 DIMENSIONS-ios（11 维）
→
判决Critical/High → BLOCK 推送
否则 PASS
→
留痕报告落 .claude/codex-audits/
设计选择原因
pre-push，不是 pre-commitCodex 审一次要几分钟；逐 commit 阻塞无法工作。pre-push 在"代码即将变公开"的那一刻审整批。
订阅版 Codex（无 OpenAI key）codex login 订阅认证，按月计费、无密钥保管。vrecorder-v2 特意不用 API key。
iOS 专属审计维度DIMENSIONS-ios.md 11 维（Swift 6 并发 / 音频会话生命周期 / 流式 partial-final / SwiftData / 权限错误分类 / 秘钥泄漏 / clean-clone 完整性…）。没照搬 vmark 的 Tauri 维度。
工具故障不硬挡超时/鉴权失败只写报告供人工复核；只有真正 VERDICT: BLOCK 才挡推送——避免审计引擎抖动卡死流水线。
可显式绕过git push --no-verify 或 SKIP_AUDIT=1 git push，但要说明理由（rule「别绕过，要问」）。
⚠️ 它和 GitHub Actions 云端审计是互补不是替代：pre-push 在本机、推送前拦截（无需云端 secret、第 1 commit 即生效）；若日后要云端兜底（PR 触发、跨机覆盖），再加 Actions——但那需要配 token secret，按需再说。

04五个阶段的提示词与启动顺序

顺序不可逆：先灌满"设计/计划"（磁盘），跑通第一件事证明闭环，最后才点火 cron。

阶段 A

设计

人类主导

rule 51 铁律：UI 只能来自 committed 设计稿，AI 不自创。先把界面设计落进 dev-docs/designs/，否则带 UI 的 WI 会被卡在 needs-design。纯后端 WI 可跳过。

提示词 · 落设计我把 claude.ai/design 的交接包放进了 dev-docs/designs/2026-06-14-v2-v1/，含每个界面的
默认/加载/错误/空四态。开发时遇到没在设计包里的界面或状态，停下并建 needs-design issue，
对应 WI 标 BLOCKED: needs-design (#N)，不要自创占位 UI。
完成判据：dev-docs/designs/<bundle>/ 已 commit，覆盖目标界面的四态。
阶段 B

计划（Gate 1 + Gate 2）

人定方向·AI 起草·独立审
1把需求写成台账行（PLANNED 五字段）
提示词 · 立项我要做这个功能：「<一句话需求>」。按 docs/features.md 的 Plan Template 整理成一行，
填 Problem/Scope/Edge cases/Test plan/Acceptance，建对应 GitHub issue 并回填 GH:#N。
（改台账时 check_gh_issue_mirror 会强制镜像，没建 issue 会被挡。）
2写实现级计划（plans/，七段）
提示词 · Gate 1 计划为 feature #N 写 dev-docs/plans/20260614-feature-N-<slug>.md，按 rule 47 七段：
Problem / Surface area(逐文件真实签名 + Files OUT) / Prior art / WI 拆分 /
Test catalogue / Risks / Backward compat。
3独立审计（Gate 2，作者≠审计者）
提示词 · Gate 2 审计用 scripts/run-codex.sh -e high 审这份计划：模型假设是否成立(字段/类型/签名真存在)、
缺哪些边界、协议签名是否泄漏实现、并发隐患、WI 拆分是否合理。零 Critical/High/Medium 才过，
最多 3 轮，不收敛就升级给我。
完成判据：plans 文档在位 + Gate-2 审计干净 → 行翻 PLANNED。
阶段 C

开发（六门）+ 提交即审计

AI 主导·你抽查

六门：Plan → 独立计划审 → TDD → 实现审 → 设备/集成验证 → 合并。第一个 feature 你先前台盯一遍，确认闭环（含提交即审计）真的通，再放手给 cron。

提示词 · 前台跑通第一件事用 /feature-workflow 实现 feature #N。严格走六门不跳门：每个 WI 先写失败测试(RED)→最小实现(GREEN)
→run-codex Gate-4 审 →行为型 WI 在模拟器/真机切片验证 →最终 WI 按 verification/SCHEMA.md 落 result:pass 证据
→版本 bump 作分支末 commit →PR 用 Refs #N。每过一门在 GH issue 追一条时间线评论。
注意:push 时 pre-push 闸门会自动审改动的 Swift——把它的结论讲给我。
门落到磁盘的产出哪个机关把关
G3 TDD测试 + 实现（<300 行、actor 隔离）rule 10/50（纪律）
G4 审计.claude/codex-audits/<branch>.mdcheck_codex_audit_artifact（merge 前必有）
pushcodex-audits/prepush-<sha>.mdpre-push 闸门（Critical/High 挡推送）
G5 验证dev-docs/verification/feature-N-<date>.mdcheck_terminal_status_evidence（没证据不准翻 VERIFIED）
G6 合并合并 + tag + 台账翻 DONE/VERIFIEDStop 钩子扫未验证 DONE / 审计欠债
完成判据：每个 WI 过六门；最终 WI 全验收 + 证据文件 → feature 翻 VERIFIED。
阶段 D

修 bug

AI 主导

bug 不走计划+计划审（是反应式的），但走实现审 + 验证门 + 提交即审计。

提示词 · 修一个 bug用 /fix-issue #N 修这个 bug：Understand → 写复现失败测试(RED) → 修(GREEN) → REFACTOR →
run-codex 审 → 验证 → 台账翻 FIXED。push 时让 pre-push 闸门复审改动。
完成判据：复现测试由红转绿；行翻 FIXED；GH issue 据验证关单。
阶段 E

无人值守（cron 闭环）

点火靠你·之后自驱

三前提：① 台账有活（阶段 B）；② 免确认授权（§02-4）；③ 闭环已前台跑通（阶段 C）。齐了才点火。

提示词 · 启动 cron 四件套用 /cron-bootstrap 按 .claude/cron-prompts/ 点火 verify/bugfix/feature/watchdog 四个 cron。
每轮：先 echo FIRED 落 .claude/cron-logs/ → 在 scope 守卫内挑一件活干完 → echo ENDED <outcome>
(work_done/no_work_in_scope/blocked/error)。没活干如实记 no_work_in_scope，别编活。
运行期间常规操作自动放行，别停下等我。
cron节奏(建议)吃什么燃料
verify每小时DONE/待验证行
bugfix每小时bugs.md TODO + GH
feature每 2 小时features.md PLANNED
watchdog每天续期 cron / 清 ghost / 查陈旧进程
⏰ cron 是会话级、7 天自动过期；关机/换会话后重新发"启动 cron"提示词即可。每轮如实落账是你早间复盘的唯一凭据。

06人类裁量权只剩三处 + 免确认

把绝大多数决策外置给规则后，只有三件事仍需要你这个人：

  1. 设计：UI 来自 claude.ai/design 的交接包，AI 不自创（rule 51）。
  2. 方向选择：需求范围、降级行为(如某能力不可用时怎么办)这类产品判断——AI 不替你拍。
  3. 真机/物理验证：需要眼睛看、手摸的验收（权限横幅、音频路由、回声），AI 跑不了。

其余全部交给：cron(何时开始) + dev-docs(记住一切) + 治理钩子 & 提交即审计(守住纪律)。免确认授权是让这套自驱起来的开关——常规工具调用预先放行，cron 会话才不会卡在权限弹窗。
